# ProductHub 压测与验收实验方案（闭环·有指标）

> 目的：用一组**可实操、有合格线、闭环复测**的实验，验证系统在目标规模下满足 SLA 务实档，并实测灾备 RPO/RTO。
> 环境：**staging（独立 ECS+RDS，灌入 ≥8000 SKU 仿真数据），绝不打 prod。** 配套脚本：[deploy/loadtest/producthub-loadtest.js](../deploy/loadtest/producthub-loadtest.js)。
> 关联：[选型决议](acceptance/选型决议-云部署.md)、[部署指南](部署指南-阿里云境内.md)、[灾备策略](数据安全与灾备策略.md)。

---

## 一、合格线（SLA 务实档 → 量化指标）
| 指标 | 合格线 | 说明 |
|---|---|---|
| 可用性 | ≥ 99.5%/月 | 由 RDS 高可用 + 监控可用率折算 |
| p95 时延·检索/详情 | < 1s | 高频核心流 |
| p95 时延·结构化检索 | < 1.5s | 多对来源叠加 |
| 错误率(非 2xx/4xx 预期) | < 0.5% | http_req_failed |
| 稳态并发 | ≥ 20 VU 稳定 | 项目目标并发 |
| RPO（数据丢失） | ≤ 15min | PITR 实测 |
| RTO（恢复时间） | 分钟级 | RDS 故障切换实测 |

> 已写入 k6 `thresholds`，跑完自动判定 PASS/FAIL。

## 二、实验清单（每条都是闭环：方法→指标→合格线→不达标处置→复测）
| # | 实验 | 方法/工具 | 关注指标 | 合格线 | 不达标处置 |
|---|---|---|---|---|---|
| E1 | 基准性能 | 1 VU 顺序打各端点，k6 | 各端点 p50/p95/p99 | 检索/详情 p95<1s | 定位慢查询(EXPLAIN)/缺索引→修→复测 |
| E2 | 阶梯负载 | k6 ramp 1→20 VU 混合场景(检索50/详情25/统计10/结构化15) | p95、错误率、吞吐、PG 连接数 | p95<1s、错误率<0.5%、20VU稳态 | 调连接池/worker、加索引、查 N+1 |
| E3 | 拐点/容量 | k6 持续加压(30/50/80 VU)至崩 | 拐点并发、资源饱和点 | 拐点 ≥ 目标并发×2(留余量) | 升规格/优化→验证选型是否够用 |
| E4 | 浸泡 soak | constant 10 VU × 2–4h + 服务端内存/连接监控 | 内存/连接是否单调增长 | 无泄漏、无慢增长 | 查连接未释放/缓存膨胀→修 |
| E5 | 峰值 spike | 瞬时 1→50 VU 再回落 | 恢复时间、错误率峰值 | 峰后<1min 恢复、不雪崩 | 加限流/排队、保护下游 |
| E6 | 大数据量 | 8000+SKU 下 深分页(page 大)、复杂多筛、全量导出 | 时延、导出耗时 | 不超时、导出有封顶提示 | 键集分页(远期)、导出分批 |
| E7 | 数据完整性·并发 | 多线程并发建同配置 SKU（脚本，非 k6） | 最终 SKU 数、指纹唯一 | 仅 1 个 SKU、无串档 | 已验证(SAVEPOINT)，回归确认 |
| E8 | 灾备演练 | RDS 控制台触发主备切换；PITR 恢复到指定时点；backup→OSS→customer-pull→restore 全链路 | RTO(切换计时)、RPO(可恢复时点) | RTO 分钟级、RPO≤15min | 调备份频率/归档、修脚本 |

## 三、执行步骤（标准闭环）
1. **准备**：staging 部署完成、灌 ≥8000 SKU、装 k6、开服务端监控（阿里云云监控/CloudMonitor 看 ECS CPU/内存、RDS CPU/连接数/慢日志）。
2. **基线**：先跑 E1 取单用户基线，作为后续对比锚。
3. **加压**：依次 E2→E3→E4→E5，每条跑完导出 k6 摘要（p95/错误率/吞吐）+ 截取同时段服务端资源曲线。
4. **专项**：E6 大数据量、E7 并发完整性、E8 灾备演练各自独立跑并记录。
5. **判定**：对照合格线逐项 PASS/FAIL。
6. **不达标 → 定位 → 修复 → 复测**（闭环，必须复测到达标）。
7. **出《压测与灾备验收报告》**，附原始数据与曲线，双方签字。

## 四、k6 用法速查
```sh
# 阶梯负载（默认到 20 VU）
k6 run -e BASE_URL=https://staging.example.com -e PH_USER=admin -e PH_PASS=*** \
       -e ROOT_TYPE_ID=26 -e SP_PAIR=3:24 deploy/loadtest/producthub-loadtest.js
# 拐点：改脚本 stages 末段 target 持续上升(30/50/80)；浸泡：改 constant-vus 10 VU × 2-4h
```
> 脚本已内置 thresholds（p95<1s、错误率<0.5%），k6 退出码非 0 即未达标，可接入 CI 门禁。

## 五、灾备演练细则（E8，承接灾备策略）
1. **RTO**：在 RDS 控制台手动触发"主备切换"，秒表计从切换开始到应用恢复可读写的时长 → 实测 RTO。
2. **RPO**：故意在 T 时刻写入一批标记数据，随后用 PITR 恢复到 T 前 1 分钟的隔离实例，核对标记数据丢失窗口 → 实测 RPO。
3. **备份链**：跑 `backup.sh`→`offsite-sync.sh`(OSS)→客户机 `customer-pull.ps1`→`restore.sh` 恢复到隔离库核行数 → 证三副本可恢复。
4. 全部实测值填入验收报告，与目标(RPO≤15min/RTO 分钟级)对照判定。
