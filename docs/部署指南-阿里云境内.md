# ProductHub 部署指南 — 阿里云境内（托管 RDS + 备案并行加速）

> 选型已定：境内·阿里云生态（ECS u1 2C4G + RDS PG16 高可用版 2C4G/100GB + OSS），~¥400-450/月，预算内双达标 RPO≤15min / RTO 分钟级 / 99.5%。
> 运营：Moore 托管。环境：staging + prod。数据：空库起步 + 迁移工具链。
> **时间策略（应对"有 deadline 越快越好"）：ICP 备案是唯一硬等待（3-22 工作日），其余全部并行填满等待期 → 总时长 ≈ 备案时长。**
> 关联：[选型决议](acceptance/选型决议-云部署.md)、[灾备策略](数据安全与灾备策略.md)、[通用部署运维指南](部署运维指南.md)。

## 0. 关键路径与并行时间线
```
Day0 ──┬─ 【关键路径】提交 ICP 备案（材料齐 3-22 工作日）───────────────► 备案通过 ─► 切公网域名上线
       ├─ 开通 ECS u1 + RDS PG16 高可用 + OSS（当天可成）
       ├─ 搭 staging、按本指南部署演练（走 IP/SSH，不需备案）
       ├─ 安全加固 + 数据迁移演练 + 压测 + 灾备首演 + 培训准备（并行）
       └─ prod 环境预搭好、只差"域名解析+签证书"这一步等备案
```
**净效果**：备案审核期内把部署/压测/迁移/灾备/培训全部做完并验收，备案一过 → 当天切公网上线。

> 加速备案的要点：Day0 即提交；主体（营业执照）、域名、网站负责人信息、核验单提前备齐；阿里云"备案小程序"可远程核验。材料清单见 §3。

## 1. 目标架构（与现仓 compose 的差异）
现仓 `deploy/docker-compose.prod.yml` 把 Postgres 跑在容器里；**境内方案改用托管 RDS**——去掉 postgres 容器，app 连 RDS 内网地址。其余（app + 前端构建 + Caddy）不变。
```
[用户] ─HTTPS─► Caddy(ECS, 自动证书) ─┬─ /api/* ─► app:8000 (FastAPI, ECS 容器)
                                      └─ /  ────► 前端静态(frontend_dist 卷)
                          app ──VPC内网──► 阿里云 RDS PostgreSQL16 高可用版（主备秒级切换）
                          备份 ──► RDS 自动备份+PITR(主) + pg_dump→OSS(异地/跨账号兜底) + 客户机拉取
```

## 2. 资源开通（Day0，约 1-2 小时）
1. **VPC + 安全组**：建专有网络；安全组仅放行 80/443（Caddy）与你的运维 IP 的 22（SSH）；**RDS 不开公网**，只允许 ECS 内网访问。
2. **ECS u1 2C4G**（华北2-北京，与用户同区降时延）：系统盘 40G+，挂数据盘可选；装 Docker + Compose。
3. **RDS PostgreSQL 16 高可用版 2C4G/100GB**：同 VPC/可用区；开**自动备份 + 日志备份（PITR）**，保留期≥7天；白名单加 ECS 内网 IP；建库 `producthub`、建专用账号。
4. **OSS Bucket**（北京，标准存储，开**版本化+生命周期**）：放加密备份/异地副本/导出件。建 RAM 子账号（最小权限：仅该桶读写）。
5. 记录内网连接串、账号密钥 → 写入 `deploy/.env.prod`（`chmod 600`，不入仓）。

## 3. ICP 备案材料清单（Day0 并行提交）
- 主体：营业执照、法人/负责人身份证、网站负责人信息与人脸核验
- 网站：域名（需在阿里云或已转入可管理）、网站名称与服务内容（如"企业内部产品报价管理系统"）
- 阿里云备案服务号（购买 ECS 满足时长即可申领）、《信息安全管理承诺书》
- 北京管局可能要求：核验单邮寄/小程序拍摄
> 备案期间网站不可对公网域名提供服务；**用 IP + 非 80/443 端口或 SSH 隧道做 staging 测试，不受影响**。备案通过后再做公网域名解析。

## 4. 部署（staging 先行，prod 同法）
### 4.1 拉代码 + 配 .env.prod
```sh
cd /opt/producthub && git clone <repo> . || git pull
cp deploy/.env.prod.example deploy/.env.prod && chmod 600 deploy/.env.prod
# 填：POSTGRES_PASSWORD(RDS 账号口令)、JWT_SECRET(openssl rand -hex 24)、SITE_DOMAIN、
#     PRODUCTHUB_DATABASE_URL=postgresql+psycopg://<acct>:<pwd>@<RDS内网地址>:5432/producthub
```
### 4.2 境内 compose 变体（去 postgres 容器，app 连 RDS）
新建 `deploy/docker-compose.aliyun.yml`（在通用版基础上删 postgres 服务、app 用外部 RDS、保留 frontend 构建 + caddy）：
```yaml
name: producthub
services:
  app:
    build: { context: ../backend }
    restart: unless-stopped
    environment:
      PRODUCTHUB_DATABASE_URL: ${PRODUCTHUB_DATABASE_URL:?set in .env.prod}   # 指向 RDS 内网
      PRODUCTHUB_JWT_SECRET: ${JWT_SECRET:?set in .env.prod}
      PRODUCTHUB_DEFAULT_CURRENCY: ${DEFAULT_CURRENCY:-USD}
      LOG_LEVEL: ${LOG_LEVEL:-INFO}
    command: sh -c "alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 2"
  frontend:
    build: { context: ../frontend }
    volumes: [ "frontend_dist:/srv/frontend" ]
  caddy:
    image: caddy:2
    restart: unless-stopped
    ports: [ "80:80", "443:443" ]
    environment: { SITE_DOMAIN: "${SITE_DOMAIN:?set in .env.prod}" }
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - frontend_dist:/srv/frontend:ro
      - caddy_data:/data
      - caddy_config:/config
    depends_on:
      app: { condition: service_started }
      frontend: { condition: service_completed_successfully }
volumes: { caddy_data: , caddy_config: , frontend_dist: }
```
> 注：RDS 已提供 HA + 自动备份 + PITR，故不需容器内 PG，也不需 pitr-setup.md 的 WAL-G 自管方案（那是无托管时的过渡桥）。
### 4.3 起栈 + 迁移 + 验证
```sh
docker compose --env-file deploy/.env.prod -f deploy/docker-compose.aliyun.yml up -d --build
# app 启动即跑 alembic upgrade head 建表
docker compose -f deploy/docker-compose.aliyun.yml logs --tail=80 app   # 确认迁移成功、JSON 日志
# staging 阶段：用 ECS 公网 IP + 非标端口或 SSH 隧道访问验证（不需备案）
```
### 4.4 初始化数据
- 空库起步：建初始管理员账号（脚本或 `users` 端点，强口令，安全交付）。
- 真实品类/属性/部件模板：由业务录入或用迁移工具链导入（见 §6）。

## 5. 安全加固（上线前必做）
- 安全组：仅 80/443 对公网；22 仅运维 IP；RDS 仅 ECS 内网白名单，**绝不开公网**。
- `.env.prod` `chmod 600`；JWT/DB 口令强随机；启动期校验拒绝 dev 占位密钥（见验收 A2-001 建议）。
- Caddy 自动 HTTPS（备案通过解析域名后自动签证书）；HSTS 可选。
- 登录限频（已内置，多 worker 共享待迭代）；导出防注入（已内置）。
- 关停后端公网直连（仅经 Caddy）；RDS SSL 连接开启。

## 6. 数据迁移工具链（空库 + 旧数据导入，承接上轮决策）
- **迁移+清洗+对账方案** + **《存量数据导入 Excel 模板》**（基于现有测试数据生成）+ **模拟导入演练**：在 staging 跑通"客户旧数据 → 导入 → 对账"全过程。
- **旧系统对接包**（旧系统 API/口径未知，预留占位）：见即将交付的《旧系统数据对接·信息索取清单与话术》。
> 这部分作为 P3 专项在 staging 演练，prod 上线时正式执行。

## 7. 压测与灾备演练（staging，合格线 = SLA 务实档）
- 压测合格线：p95 检索/详情 <1s、可用性链路通、并发 ≤20 稳定（详见即将交付的《压测与验收实验方案》）。
- 灾备首演：触发 RDS 故障切换计时（RTO）、PITR 恢复到指定时点（RPO）、`offsite-sync.sh`→OSS、`customer-pull.ps1` 拉取、`restore.sh` 恢复演练。记录实测 RPO/RTO 对照目标。

## 8. prod 上线切换（备案通过后）
1. 域名解析指向 ECS 公网 IP；Caddy 自动签发证书。
2. prod 起栈（同 §4，独立 .env.prod 与 RDS 实例/库）。
3. 冒烟回归（登录/检索/详情/配置/录价/报价/导出全链路）。
4. 灾备脚本 cron 装上（backup→OSS、offsite-sync、watchdog、customer-pull）。
5. **go-live 验收 checklist + 双方签字** → 正式放量。
6. 回切预案：保留切换前快照/备份；异常则解析回退 + 数据从 PITR/备份恢复（见部署运维指南 §4）。

## 9. 上线验收 checklist（节选）
- [ ] HTTPS 正常、证书有效；后端不可公网直连
- [ ] RDS 高可用开启、自动备份+PITR 生效（pg_stat / 控制台确认）
- [ ] 首次备份成功 + 一次恢复演练通过（实测 RPO/RTO 达标）
- [ ] 监控告警可达（WEBHOOK_URL 通）、值班就绪
- [ ] 全链路冒烟通过；初始账号交付（安全渠道）
- [ ] 双方签字、运维手册与用户手册交付
