### 🟢 A4 数据完整性红线 = 实质零缺陷（最重维度）
对抗复核逐条 confirmed 或下调（复核员判定"P0 只应给未缓解漏洞，这些是确认正确"）：
- 指纹 `fingerprint` 唯一约束（DB uq + ORM unique）、规范化序列化仅用不可变 `code`、属性按 code 排序、子件按 slot.code 排序 → 同配置必同指纹。
- **黑盒序列化仅含 `part.code` 不含供应商** → 改供应商指纹零变化（红线成立）。
- 白盒供应商 `|S:supplier.code` 仅在标注时追加，既有 SKU 指纹逐字节零变化。
- `sku_code` 原子自增、写一次永不改；**改配置 = create_sku 新建 + 旧 SKU superseded 留痕，绝不原地改指纹/编码**（M2 治理闭环）。
- 复合 FK（slot×type、part×type、attr×option）DB 层强制类型匹配；并发同指纹用 SAVEPOINT + IntegrityError 兜底 → 库内仅 1 个 SKU。
- 迁移 downgrade 抽查 3 条全可逆，无孤表/列/约束。
- **结论：业主"数据库绝不能数据混乱""指纹一经生成永不原地修改"红线，代码层面站得住。**

### 🟡 校准后两条真 P1（待接受风险，非阻断）
1. **A3-F1 SKU 详情 N+1**：`skus.py` `_node_out` 每节点 `db.get(NodeType)` ×2 + `attribute/option/purchased_part.supplier/slot` 关系未预加载，详情递归触发级联懒加载（20 节点 BOM → 60+ 次 DB 往返）。高频路径（详情抽屉），confirmed。**修法**：`detail()` 加 `selectinload` 链 + 预缓存 NodeType 字典。
2. **A6 P1-001 前端零测试**：无 vitest/playwright，主链路 UI（配置/报价/权限反馈）无测试锁定。confirmed。后端有 74 测试覆盖红线，前端缺口为待补。

### P1→P2 被对抗复核下调（值得记、非严重）
- **A2-001 JWT dev 密钥**：`.env` 已 gitignore 且无 git 历史；prod 用 `docker-compose.prod.yml` 的 `${JWT_SECRET:?}` 强制外部注入 → 仅 dev 范围风险。建议加启动期"拒绝 dev-only-* 密钥"校验。
- **A2-002 报价单 IDOR**：早期版本曾存在，**已在 commit 4a0dcdd 修复**，9 个 quote_id 端点全调 `_own_quote_or_404`。下调因缺跨用户回归测试 + 单点防护。
- **A2-003 CSV 公式注入**：导出层 `_csv_safe` 已护报价主字段；存储侧 part/template 名未过滤（若经其它导出口可能漏）。
- **A3-F2 连接池余量**：2 worker×30=60 vs PG 100，余 40；但同步短连接、实际并发远低于理论值。
- **A3-F3 offset 深分页**：仅 `sort=name`（无 btree 索引）与 `price`（相关子查询）受影响；`recent/code` 走索引相对安全。
- **A5-001 JWT 工程流程**：同 A2-001，gitignore+prod 注入已缓解。

### P2 上线前硬化建议（不卡门槛）
- **安全**：A2-004 LIKE 通配符未转义；A2-005 登录限频仅进程内（多 worker 不共享）；A2-006 IntegrityError 错误信息可能泄露约束名。

- **架构**：A1-001 价格逻辑嵌 API（与导入路径各写一份）应抽 `price_service`；A1-003 报价逻辑 + parts 原生 SQL 应 service 化；A1-002 前端 ~80+ 处 `any`；A1-004 递归组件 props 缺类型。

## Gate 2 业务线 — 复盘（扮演真实用户跑高/中/低频与跨角色场景）

方法：进程内 TestClient + 真实 JWT（`make_access_token`，端到端验真实鉴权链+RBAC）。**读场景走 dev 库（只读），写场景走一次性克隆库 `producthub_accept`（`TEMPLATE producthub` 秒级克隆，跑完即弃，零污染 dev）。** 因 Gate1 工作流压崩内存（仅 1.59G 余），改为内联串行执行、不堆并发。

### 写场景（克隆库）：32 / 32 PASS
- **C1 建 SKU + 去重 + 指纹不可变**：提交原配置→去重命中既有(created=False, 同指纹同编码)；改一个属性→新指纹(异配异指纹, 新建 SKU-2026-25004)；同新配置二次提交→去重稳定；**原 SKU 落库指纹/编码绝未原地改（红线 PASS）**。
- **C2 录价 + 同日纠错 supersede**：录入 123.45→现价含之；同日再录 130.00→现价更新、同币种现价唯一、旧价 supersede 灰显。
- **C3 RBAC**：sales（无录价权）录价 → **403**。
- **C4 报价单 E2E**：建单→加明细→导出 xlsx(200)→**导出后改明细被冻结 409**。
- **C5 跨用户 IDOR**：sales 读/改 admin 的报价单 → **404**（横向越权防护有效）。
- **C6 SKU 作废/恢复**：retire→retired，restore→active。
- **C7 部件改名 resync**：改名→引用 SKU 摘要名实时同步为新名，**指纹/编码不变（红线 PASS）**。

### 读场景（dev 库只读，API total 对 SQL 真值）：29 / 30 通过（1 条系验收脚本断言写错，产品经 SQL 证实正确）
- **R1-R4 全 PASS**：全量 total=8003、active=7006、按品类、单规格(option_id)、**结构化检索 sp_pair 黑∪白(508)**、供应商黑∪白过滤(919) 均与 SQL 真值逐一对齐。
- **R5（脚本断言错，非缺陷）**：`category-breakdown sum=982 ≠ 供应商 active 总量 919`。SQL 坐实：sum_over_types=982、distinct=919，差 63 来自"一个 SKU 同时用该供应商供多个件类型"。这是用户当初设计的"件目录规模"口径的应然结果（计数=用到该供应商该件类型的在售 SKU 数），**下钻"对上"已 PASS（件类型 count 9 == /skus 过滤 9）**。我的断言误设了 sum==distinct 不变式。
- **R6 详情+BOM+价格**、**R7 stats/overview**、**R8 全部 sort/币种/区间/quotable/sourcing(blackbox/whitebox/direct)** 均 200。
- **R9 边界**：非法 status/sort/currency → **422**；深翻页(page 99999) → 优雅 200 空页；不存在 SKU → **404**；未带令牌 → **401**。

## Gate 3 交叉评审（技术发现 ↔ 业务影响对账）

- **A3-F1 SKU 详情 N+1 ↔ B1 高频详情流**：详情抽屉是日常高频操作，N+1 直接拖慢它 → **此 P1 的优先级上调，建议上线前修**（修法低成本：`detail()` 加 `selectinload` 链 + 预缓存 NodeType）。
- **A3-F3 offset 深分页 ↔ 业务**：业务默认 page_size=20，且有分面/结构化检索收窄结果，用户极少深翻页 → 实际影响低，P2 维持。
- **A2 安全 P2 群 ↔ 业务 RBAC**：Gate2 已实测 RBAC 生效（sales 录价 403、跨用户报价 404）→ A2 残留项均为纵深防御（密钥卫生、CSV 存储侧、LIKE 转义、限频多 worker），非业务流断裂。
- **A5 运维缺口 ↔ 部署交付**：前端缺 Dockerfile、部署 runbook/回滚预案缺失 → 非产品 P0，但**是平滑交付运维的真实前置债**，建议上线前补。
- **数据完整性（A4）↔ 业务写场景（Gate2 C1/C7）**：代码审计"指纹不可变/改名 resync"与业务端到端实测互证——红线双向确认无缺口。
