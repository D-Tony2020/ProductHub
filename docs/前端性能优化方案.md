# ProductHub 前端性能优化方案（面向 2C2G + 3M 带宽 + 移动端）

> 目标：在**不换组件库（Element Plus 保留）、不引重型依赖、不破坏功能、不重写业务逻辑**前提下，把首屏 JS/CSS 的 gzip 体积从约 **280–360KB 降到约 120–170KB（-50%~-60%）**，并消除每页 325KB 大图与无缓存的浪费——3M 固定带宽下首屏关键资源下载预估改善 **1–2 秒**，移动端业务员可交互时间明显提前。
> 方法：5 维静态审计（打包/路由/渲染/网络/资源）+ 跨维汇总。体积数字为基于依赖构成与 Element Plus 公开体积的**工程估算**（本机内存受限未跑 build），上线以构建产物真实 `.br` 大小 + DevTools Network 瀑布复核。

---

## 一、现状根因（诊断）
1. **Element Plus 全量引入**：`main.ts:1` `import ElementPlus` + `:13` `app.use(ElementPlus)` → 注册全部 ~80 组件，全站实际只用约 30 种，未用的 ~50 个无法 tree-shake，全进首屏。
2. **整库 CSS 单入口**：`styles/index.scss:21` `@use 'element-plus/theme-chalk/src/index.scss'` 拉入整库 CSS（render-blocking）。
3. **logo-square.png 1127×1127 / 325KB**，`MainLayout` 每页常驻、实渲 34px → 每次新访问白下 ~320KB。
4. **Caddyfile 无 Cache-Control**：hash 资源无法强缓存，切页/二访走一串 304 往返。
5. **vite 裸配**：无 manualChunks（vendor 与业务混编，发版即缓存全失效）、无预压缩、未锁 build target。
6. **首屏页拖入低频重组件**：`SkuList.vue:14` 静态 import `StructuredSearchDrawer`（链式拖入 `useConfigurator` + 递归 `ConfigSearchNode`）。
7. **无移动端适配**：固定 210px 侧栏 / 360px 搜索框 / 5 列 stat-band / 1100px+ 表格 → 手机横向溢出。
> 正面确认（**保持不动**）：路由已全部懒加载（`router/index.ts` `()=>import()`）；图标已全部具名导入（tree-shake 友好）；字体是系统字体栈（无 webfont 下载）。

## 二、速赢清单（按"性价比"排序，先做）
| # | 动作 | 预期收益 | 工作量 | 改动面 |
|---|---|---|---|---|
| 1 | **logo 换小图/SVG**：把 1127² PNG 缩到 64–96px 或导出 SVG（3–8KB），替换 `assets/logo-square.*` | 每次新访问省 ~**320KB**；3M 下省近 1s | **S** | 1 个文件（图工具处理，不跑 build） |
| 2 | **Caddy 长缓存头**：`/assets/*` 设 `Cache-Control: public,max-age=31536000,immutable`，`index.html` 设 `no-cache` | 消除切页/二访的 304 往返串；移动高 RTT 下切页明显变快 | **S** | 仅 `deploy/Caddyfile` |
| 3 | **Element Plus 改按需**：`unplugin-auto-import` + `unplugin-vue-components` + `ElementPlusResolver`；`main.ts` 删全量、locale 转 `<el-config-provider :locale="zhCn">` | 首屏 EP 组件 JS gzip ~130–180KB → ~**70–100KB**（摇掉 ~50 未用组件） | **M** | `main.ts`、`vite.config.ts`、`App.vue`、`package.json` |
| 4 | **SkuList 抽屉组件级懒加载**：`StructuredSearchDrawer`/`BomTreeNode`/`PriceTrendChart` 改 `defineAsyncComponent`，模板加 `v-if`（当前 v-model 常驻必须加 v-if 才真延迟） | 配置器+递归节点移出首屏，业务 JS 再降 ~15–30KB gzip | **S** | `SkuList.vue:10,11,14,672…` |
| 5 | **预压缩 .br/.gz + Caddy precompressed**：`vite-plugin-compression` 构建期生成 `.br`/`.gz`；Caddy `file_server { precompressed br gzip }` 直接吐静态 .br | 文本资源再省 15–25%，且 2C2G **CPU 零消耗**（停实时压缩） | **S** | `vite.config.ts`、`package.json`、`Caddyfile` |
| 6 | **vite manualChunks 固定 vendor 分包**：vue/pinia→`vue-vendor`、element-plus→`element-plus`、其余→`vendor` | 发版只重下变动业务 chunk，~150KB+ EP/Vue 长缓存命中 | **S** | `vite.config.ts` + Caddy 长缓存(同#2) |

> ⚠️ brotli 纠正：标准 `caddy:2` 镜像**不能运行时 `encode br`**（需 xcaddy 自定义构建）；用 #5 的"预生成 .br + `file_server precompressed`"即可在标准镜像上吐 brotli，无需换镜像。

## 三、完整方案（按优先级）
**P1 · 首屏体积削减（最大杠杆，先做）**
- 速赢 #1 logo、#2 缓存头、#3 EP 按需、#4 抽屉懒加载。
- CSS：本场景**保留单入口全量 CSS**（`index.scss` 的 `@forward var.scss` 编译期主题梯度 primary #1f4e79 light-3..9 依赖单一 SCSS 入口；切组件级 `importStyle:'sass'` 会与单入口覆盖冲突，DESIGN.md 已记此坑），靠 #5 brotli 把 50–70KB gzip 压到 ~35–50KB。组件级 CSS 按需作为后续单列任务（需手测配色）。

**P2 · 二级页瘦身与缓存 / 移动端**
- 速赢 #5 预压缩、#6 manualChunks。
- `MainLayout` 空闲预取：`requestIdleCallback` 预取高频二级页（Configure/Quotes），弱网（`navigator.connection.saveData/effectiveType`）跳过、`try/catch` 包裹不阻塞首屏。
- `SkuList` 回写 URL 的 `deep watch`（:324–327 对整棵 filters 深比较）改监听 `buildQuery()` 序列化结果 + 200ms 防抖，缓解低端机输入/勾选迟滞。
- **移动端响应式**（业务员手机主路径，渐进增强、不改业务逻辑）：`MainLayout` 加 `@media(max-width:768px)` 侧栏默认折叠/转抽屉、搜索框 `max-width:100%`；`SkuList` stat-band 窄屏降 `repeat(2,1fr)`、工具条 `flex-wrap:wrap`；密集表格窄屏优先用已有卡片栅格或容器 `overflow-x:auto` 兜底。先做 MainLayout+SkuList。

**P3 · 构建稳健性与边角（确认/小修）**
- `vite build` 显式锁定 `{ sourcemap:false, target:'es2018', minify:'esbuild', cssCodeSplit:true }`（es2018 兼顾旧移动端 WebView；esbuild 省内存契合 2C2G 构建）。
- `Login.vue:30` 卡片 `width:380px` → `width:100%;max-width:380px`（窄屏不溢出）。
- 清理本地 `frontend/dist` 陈旧 hash 产物（仅卫生问题，生产走容器重建不受影响）。
- PartsTable 一次性渲染 200 行 el-table（非首屏，进供应商页时的运行时重负载）→ 后续单列"分页/虚拟滚动"任务。

## 四、预期收益
- 首屏 JS/CSS gzip：约 **280–360KB → 120–170KB（-50%~-60%）**；叠加 logo -320KB、缓存命中省 304。
- 3M 固定带宽（移动端实测常 100–200KB/s）下首屏关键资源下载预估改善 **1–2s**；2C2G 主线程少解析一大块 JS，TTI 提前。
- 回头客/业务员每次发版省 ~150KB+ EP/Vue 重复下载（vendor 长缓存）。
> 均为工程估算，上线后以 `dist` 真实 `.br` 大小 + DevTools Network 瀑布复核。

## 五、破功能护栏 + 回归手测（按需迁移有真实破功能陷阱，必做）
1. **保留组件库与主题**：仅按需引入 + 单入口全量 CSS，不换 EP、不动 `index.scss` 编译期主题梯度。
2. **v-loading（7 处）**：`ElementPlusResolver({ directives:true })`（默认开，勿关）；手测 SkuList:684/726、PartsTable:199、PartDetailDrawer:122、PartPicker:104、AdminImport:67、Suppliers:267 遮罩仍显示。
3. **ElMessage/ElMessageBox（13 处）**：**保留现有具名 import**（含纯 .ts 的 `api/client.ts:2`、AdminUsers.vue:33 动态 import 变体），不交给 auto-import，零体积代价、最稳；逐个触发确认弹窗正常。
4. **locale**：`<el-config-provider :locale="zhCn">` 包裹后，分页器/日期/确认框中文文案仍生效。
5. **异步组件**：`defineAsyncComponent + v-if` 后抽屉首次打开能拉取并渲染、二次状态正常；递归组件 `BomTreeNode`/`ConfigSearchNode` 异步化后内部自引用（按 name 解析）不报未注册。
6. **构建后冒烟**：`npm run build && npm run preview` → 登录→货架(卡片/表格双视图/统计带/品类树)→SKU 详情抽屉(BOM 递归树+价格图)→按配置找货抽屉(配置器可用)→看板/报价/供应商切换无白屏报错。
7. **Caddy precompressed**：验证响应头 `Content-Encoding: br` 且页面正常加载（无解码失败）。
8. **移动端**：375px 窄屏首屏布局无错位/横向溢出。

## 六、实施建议
分两批：**第一批速赢**（#1 logo + #2 缓存头 + #5 预压缩/precompressed + #6 manualChunks）——多为配置/资源改动、风险低、收益快、可先上单机试用看效果；**第二批 EP 按需 + 抽屉懒加载**（#3#4，有破功能风险，按第五节护栏逐点手测后再合）。移动端响应式（P2）作为独立批次。

## 七、节奏：部署后优化（backlog）+ 待建 CI 门禁
**决策**：前端优化整体**压到部署后**做（先上单机试用、看真实使用再优化）。本方案上文 P1/P2/P3 即"留着的 backlog"，已带工作量/文件/回归护栏，届时按批次领取。

**辨析（重要）**：这些是**改动任务**，不是"测试任务"。真正进 CI/CD 的是下面的**性能门禁**（CI 测试任务）——优化落地时自动验证达标、平时防回退：
| CI 门禁（待 CI 流水线建好时加） | 作用 |
|---|---|
| `vite build` 必过 | 阻断构建失败合入 |
| `vue-tsc --noEmit` 类型检查 | 现 build 不含类型检查，补上防类型回归 |
| 首屏体积预算（如 size-limit / 对 dist 主 chunk 断言阈值） | 超阈值 fail，把"首屏 gzip ≤170KB"钉成硬门禁 |
| （可选）Lighthouse CI 性能分阈值 | 端到端体验回归守门 |
> 现状：仓库**暂无 CI/CD 流水线**（无 `.github/workflows` 等）。CI 属"待建"，要持续集成时再搭；在此之前优化项就以本文档为 backlog 留存。

**建议仍在部署时顺手做的 2 项（部署配置层、零风险、非"优化"压后范畴）**：① logo 换小图（速赢 #1，每页省 ~320KB）；② Caddy 缓存头（速赢 #2，仅改 Caddyfile）。其余压后合理。
