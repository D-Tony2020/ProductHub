# ProductHub 单机自管部署方案（测试期 · 自管 PITR + 资源治理）

> 路线（业主定）：先上**单机**做测试；自动备份 + 时点恢复**自己运维**；做好性能与资源管理。
> 定位：一台机器同时跑 应用 + 数据库，省掉托管 RDS 的钱（约 ¥350/月）。**代价是无自动高可用**——机器宕机=停服，靠备份恢复（数十分钟），见 §8 风险与升级路径。
> 配套文件：[postgres-pgbackrest.Dockerfile](../deploy/postgres-pgbackrest.Dockerfile)、[docker-compose.single.yml](../deploy/docker-compose.single.yml)、[pgbackrest.conf.example](../deploy/pgbackrest.conf.example)、备份看门狗 [backup-watchdog.sh](../deploy/backup-watchdog.sh)。

---

## 1. 主机规格（单机）
| 项 | 推荐 | 说明 |
|---|---|---|
| 规格 | **2核8G**（最低 2核4G） | app+DB+前端构建同机，8G 更稳；数据仅 51MB，DB 不吃内存，瓶颈是"同机共挤" |
| 系统盘 | 40-50GB SSD(ESSD) | 系统 + 镜像 |
| 数据盘 | 另挂 40GB SSD（可选但推荐） | PG 数据 + pgBackRest 本地仓 IO 与系统隔离 |
| Swap | **2-4GB**（必配） | 兜住前端构建/导出的瞬时峰值，防 OOM；`vm.swappiness=10` |
| 系统 | Ubuntu 22.04 / Alibaba Cloud Linux | 装 Docker + Compose v2 |
> 月费量级：境内轻量/ECS 2核8G ~¥50-90/月（待计算器核）。比"应用+托管RDS"省约 ¥300/月。

## 2. 架构与组件（同机）
```
[用户] ─HTTPS─► Caddy(自动证书) ─┬ /api/* ─► app:8000 (FastAPI, 2 worker)
                                 └ /  ────► 前端静态(frontend_dist 卷)
   app ──容器内网──► postgres:5432 (PG16 + pgBackRest)
   pgBackRest ──► 连续WAL归档 + 全量/增量备份 → 本地仓(+强烈建议推 OSS 异地)
```
- 全部容器加**资源上限**（compose 已设：postgres 1.5C/2G、app 1C/1G、caddy 0.5C/256M），防 app 与 DB 互抢、防单容器吃爆整机。
- PG 已按小机调优（compose `-c` 注入，见 §4）。

## 3. 部署步骤
```sh
cd /opt/producthub && git clone <repo> . || git pull
# 配密钥与 pgBackRest 配置
cp deploy/.env.prod.example deploy/.env.prod && chmod 600 deploy/.env.prod   # 填 POSTGRES_PASSWORD/JWT_SECRET/SITE_DOMAIN
cp deploy/pgbackrest.conf.example deploy/pgbackrest.conf                       # 本地仓即可起步；建议启用 OSS 块
# 起栈（首启构建镜像、跑迁移、签证书）
docker compose --env-file deploy/.env.prod -f deploy/docker-compose.single.yml up -d --build
# 初始化 pgBackRest stanza（一次性）
docker compose -f deploy/docker-compose.single.yml exec postgres pgbackrest --stanza=producthub stanza-create
docker compose -f deploy/docker-compose.single.yml exec postgres pgbackrest --stanza=producthub check   # 校验归档通路
# 首个全量备份
docker compose -f deploy/docker-compose.single.yml exec postgres pgbackrest --stanza=producthub --type=full backup
# 验证 + 冒烟
docker compose -f deploy/docker-compose.single.yml exec postgres pgbackrest --stanza=producthub info
curl -fsS https://<SITE_DOMAIN>/healthz
```
初始管理员账号用强口令创建并安全交付；真实模板/数据走迁移工具链（见对接包+导入模板）。

## 4. PostgreSQL 调优（小机口径，compose 已注入）
- `shared_buffers=1GB`、`effective_cache_size=2GB`（数据 51MB → 全量+索引常驻内存，查询走内存、IO 极低）
- `work_mem=16MB`、`maintenance_work_mem=256MB`、`max_connections=100`
- `max_wal_size=2GB / min_wal_size=256MB`、`random_page_cost=1.1`、`effective_io_concurrency=200`（SSD）
- 归档：`wal_level=replica`、`archive_mode=on`、`archive_command=pgbackrest ... archive-push %p`、`archive_timeout=60`（即使低写也每 60s 切段 → **RPO 上限 ~60s**）
- **连接对齐**：app 2 worker × (pool 10 + overflow 20)=60 ≤ PG 100，余 40 给备份/维护。**勿把 worker 调高致总连接 >80**。

## 5. 自管 PITR + 备份（pgBackRest）
**节奏（host crontab，经 docker compose exec 调用）：**
```sh
# 每周日 01:00 全量；周一至六 01:00 增量
0 1 * * 0  cd /opt/producthub && docker compose -f deploy/docker-compose.single.yml exec -T postgres pgbackrest --stanza=producthub --type=full backup
0 1 * * 1-6 cd /opt/producthub && docker compose -f deploy/docker-compose.single.yml exec -T postgres pgbackrest --stanza=producthub --type=incr backup
# 每日 09:00 备份看门狗（心跳过旧告警）；建议装在另一台/客户机
0 9 * * *  WEBHOOK_URL=<机器人> /opt/producthub/deploy/backup-watchdog.sh
```
> 备份成功后写心跳供看门狗判活；pgBackRest `info` 可看各备份与 WAL 连续性。

**时点恢复（演练 / 真实灾难）：**
```sh
# 恢复到"崩溃前任意一秒"：停应用→PITR 还原→起库重放到目标时点
docker compose -f deploy/docker-compose.single.yml stop app caddy
docker compose -f deploy/docker-compose.single.yml exec postgres \
  pgbackrest --stanza=producthub --type=time "--target=2026-06-20 14:30:00+08" --delta restore
docker compose -f deploy/docker-compose.single.yml restart postgres   # PG 重放 WAL 到目标点
docker compose -f deploy/docker-compose.single.yml start app caddy
```
**每月恢复演练**（强制）：在隔离实例/目录跑一次 PITR + 核行数，实测 RPO/RTO 并记录——**未演练的备份按没有备份对待**。

## 6. 资源与性能管理（单机关键）
1. **容器上限**（compose 已设）：防 app 忙时拖垮 DB、防单容器 OOM 拖死整机。
2. **Swap 必配 2-4GB**：兜前端 vite 构建（~1-2GB）与 Excel 导出的瞬时峰值。
3. **前端构建内存坑**：vite 构建吃内存，2核4G 同机构建可能 OOM → 选项：① 先配足 swap；② 低峰构建；③ **在开发机预构建镜像后 `docker save`/推私有仓再 pull**（最稳）。
4. **磁盘/WAL 监控（自管 PITR 头号风险）**：若归档持续失败（OSS 密钥错/磁盘满），WAL 会堆在 `pg_wal` → 撑爆磁盘 → **PG 直接停**。已设 `archive-push-queue-max=2GiB` 作安全阀；**必须监控并即时告警**：`pg_stat_archiver.failed_count`、`pg_wal` 目录大小、磁盘使用率。
5. **基线测量**：上线后用 `docker stats` + PG `pg_stat_*` 测实际 CPU/内存/连接/缓存命中，按真实值微调上限与 `shared_buffers`。

## 7. 监控与告警清单（轻量自管）
| 指标 | 来源 | 告警阈值 |
|---|---|---|
| 磁盘使用率 | 主机 | >80% |
| 内存/Swap 使用 | 主机/docker stats | 持续 >85% / Swap 频繁换入换出 |
| 归档失败数 | `pg_stat_archiver.failed_count` | >0 立即告警 |
| pg_wal 目录大小 | 容器内 | 异常增长 |
| PG 活跃连接 | `pg_stat_activity` | >80 |
| 缓存命中率 | `pg_stat_database` | <99%（小数据应近 100%） |
| 备份心跳 | backup-watchdog.sh | 超 26h 无成功备份 |
| 容器存活/healthz | docker / curl | 异常 |
> 可用阿里云云监控 agent + 一个轮询 `pg_stat_archiver`/磁盘的小脚本接 WEBHOOK_URL（企业微信/钉钉/飞书）。

## 8. 风险与升级路径（诚实）
- **无自动高可用**：机器宕机即停服，RTO = 重建+恢复时长（数十分钟）；PITR 保 RPO ≤60s（数据基本不丢），但**可用性靠你接受"宕机后恢复"**。
- **本地仓单点**：本地备份仓随机器损毁同归于尽 → **强烈建议启用 pgBackRest 的 OSS 仓**（异地），单机也要有异地副本。
- **自管心智负担**：归档失败、磁盘、恢复演练都要自己盯（§5/§6/§7 已给护栏）。
- **升级路径（业务依赖加深后）**：平滑迁到"应用 + 托管 RDS 高可用"——RDS 开 PITR+多AZ，自管 pgBackRest 退役；数据用 `pg_dump`/逻辑复制迁入。本单机方案不是死胡同，是低成本起步 + 验证。
