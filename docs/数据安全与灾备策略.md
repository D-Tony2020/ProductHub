# ProductHub 数据安全与灾备策略

> 课题来源：本产品长期维护客户的报价与产品信息，**数据丢失或灾难性停机将造成重大损失**。
> 本文定目标、威胁模型、3-2-1 备份与 DR 路线、运维 SOP，并给出落地脚本/配置清单。

## 1. 已定目标（业主决策）
| 指标 | 目标 | 含义 |
|---|---|---|
| **RPO**（最多丢多少数据） | **≤ 15 分钟** | 需 WAL 持续归档（PITR）；纯日备份的 24h 不达标 |
| **RTO**（最多停多久） | **分钟级** | 需热备/多 AZ 自动切换；单机人工恢复不达标 |
| **数据库形态（未来）** | **托管 Postgres** | 阿里云 RDS / 腾讯云等，自带自动备份+PITR+多 AZ |

> 核心结论：**RTO 分钟级 + RPO≤15min 的最优解 = 迁移到托管 Postgres**（云厂商内置 PITR + 多 AZ）。
> 故策略分两阶段：**过渡期单机把风险压到可接受**，**目标态迁托管把灾备交给云厂商**。

## 2. 威胁模型（要防什么）
1. **数据丢失**：磁盘损坏、误删、`DROP`、勒索加密、机房/区域故障。
2. **灾难性停机**：主机宕、Docker/PG 崩、区域中断 → 业务不可用。
3. **逻辑损坏/串数**：bug/并发导致数据错乱（≠ 丢失）。
4. **泄露**：备份含报价金额/客户名等商业敏感数据，离机即需加密。
5. **静默失败**：备份"以为在跑实则早停"——最危险，因为出事才发现。

## 3. 现状 vs 目标
| 维度 | 现状 | 目标 |
|---|---|---|
| 逻辑备份 | 日 `pg_dump` 30 天 | 日备份 + GFS（日30/周12/月12）✅本轮已升级 |
| RPO | 24h | ≤15min（PITR/WAL 归档 或 托管库内置） |
| 异地副本 | 仅口头备注 | 对象存储加密 + 客户本地机加密副本（3-2-1）✅本轮已落脚本 |
| 监控告警 | 无 | 备份失败/体积骤降即告警 + 心跳看门狗 ✅本轮已落 |
| 恢复演练 | 无 | 每月演练到隔离库 ✅本轮已落 restore.sh+SOP |
| RTO/HA | 单机 SPOF | 托管多 AZ 自动切换（迁移后） |
| 逻辑损坏防护 | 指纹不可变+约束+审计（验收已确认） | 维持 + PITR 可回退到损坏前时点 |
| 文件件（导出/上传） | 落本地盘随机器丢 | 外置对象存储（无状态化） |

## 4. 3-2-1 备份原则（落地映射）
- **3 份副本**：① 生产库（live）② 本机每日 dump（`backup.sh`）③ 异地（对象存储 + 客户本地机）。
- **2 种介质/位置**：云主机本地盘 + 对象存储/客户 PC。
- **1 份离线异地**：客户办公机 `customer-pull.ps1` 拉取的加密副本（跨厂商兜底）。

## 5. 落地清单（本轮交付的脚本/配置）
| 文件 | 作用 | 状态 |
|---|---|---|
| `deploy/backup.sh` | 日全量 + GFS 保留 + 失败/体积告警 + 成功心跳 | ✅已增强 |
| `deploy/backup-watchdog.sh` | 心跳过旧告警，兜住"cron 整个没跑"的盲区（建议装异地/客户机） | ✅新增 |
| `deploy/offsite-sync.sh` | rclone crypt 推对象存储（加密+版本化） | ✅新增 |
| `deploy/customer-pull.ps1` | 客户机定时拉【加密】副本（数据握在自己手里，解密密钥另存） | ✅新增 |
| `deploy/restore.sh` | 恢复演练到隔离库（验证备份可用） | ✅已有 |
| `deploy/pitr-setup.md` | WAL-G 过渡期 PITR（RPO→分钟级），迁托管后退役 | ✅新增 |

> 说明：以上为**生产主机/客户机的部署期配置**，未在开发机实跑（无对象存储凭据/生产 PG）。
> 上线时按各文件前置步骤配置并跑**验证项**（pg_stat_archiver、恢复演练）后方算生效。

## 6. 运维 SOP（节奏与责任）
| 任务 | 频率 | 命令/动作 |
|---|---|---|
| 日全量备份 | 每日 02:30 | cron `backup.sh` |
| 基础备份（PITR） | 每日 01:00 | cron `wal-g backup-push`（见 pitr-setup.md） |
| 异地同步 | 每日 03:00 | cron `offsite-sync.sh` |
| 客户机拉取 | 每日 | 任务计划 `customer-pull.ps1` |
| 备份看门狗 | 每日 09:00 | cron `backup-watchdog.sh`（异地/客户机最佳） |
| **恢复演练** | **每月** | `restore.sh` 到隔离库 + 记录实测 RPO/RTO |
| 密钥/凭据保管 | 持续 | rclone.conf / WAL-G / crypt 密钥 chmod 600、离线另存（否则备份无法解密=废） |
| 告警通道 | 一次性 | 设 `WEBHOOK_URL`（企业微信/钉钉/飞书机器人） |

## 7. 路线图
- **第一阶段（现在·单机过渡，1–2 周内）**：装 cron 跑 backup/offsite/watchdog；配 WAL-G PITR；客户机配 customer-pull；跑首次恢复演练。→ RPO 24h→分钟、异地三副本、失败可感知。RTO 仍数十分钟（人工），可接受过渡。
- **第二阶段（目标态·迁托管 Postgres）**：选定云厂商托管库，开自动备份+PITR+多 AZ；导出/上传件外置对象存储；App 无状态多实例。→ RPO≤15min、**RTO 分钟级（自动切换）**达标。本阶段后单机自管 PITR 退役，仅保留逻辑 dump 异地副本作跨厂商兜底。

## 8. 红线提醒
- 备份/同步密钥一旦丢失，加密备份**永久不可恢复**——密钥必须离线另存、与备份分离保管。
- 恢复进生产库会覆盖现有数据，仅在确认损坏且演练通过后执行（见 docs/部署运维指南.md）。
- 任何"已配置"都必须有**验证证据**（归档计数增长、演练恢复成功）才算数——未验证的备份按"没有备份"对待。
